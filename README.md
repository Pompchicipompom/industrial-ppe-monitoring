# Система событийного контроля ношения средств индивидуальной защиты по видеопотоку

## Краткое описание

Проект реализует программный модуль для автоматического контроля ношения
средств индивидуальной защиты (СИЗ) по данным видеонаблюдения. Система
анализирует видеопоток, обнаруживает человека и элементы СИЗ (каску,
жилет), связывает их с конкретным человеком, подтверждает нарушение во
времени и формирует журнал событий, пригодный для разбора инцидентов.

Репозиторий содержит только итоговую программную часть выпускной
квалификационной работы. Экспериментальные скрипты, baseline-методы,
исторические запуски и черновые отчёты в репозиторий не включены.

## Что делает система

- обработка видеофайла или видеопотока (включая RTSP);
- обнаружение человека и элементов СИЗ;
- привязка каски к области головы;
- привязка жилета к области торса;
- временное подтверждение нарушения (только устойчивое отсутствие СИЗ);
- подавление повторных событий внутри одного эпизода (cooldown);
- консолидация повторных срабатываний через `EventConsolidatorV3`;
- формирование `events.csv` / `events.jsonl` и итогового
  `consolidated_events.csv`;
- визуализация результата на видео (`processed.mp4`).

## Научно-техническая идея

Основная идея проекта состоит в переходе от покадровой детекции объектов
к событийной фиксации нарушений по конкретному человеку в видеопотоке.
Нарушение формируется только после временного подтверждения по треку
человека и последующей консолидации повторных срабатываний, возникающих
из-за смены идентификатора трека и кратковременных пропусков детекции.

## Архитектура

```
видео (файл / RTSP)
    │
    ▼
семплинг кадров (motion-gated)
    │
    ▼
детектор (person / head / hardhat / vest / no_vest)
    │
    ▼
person tracker (стабильные track_id)
    │
    ▼
ассоциация СИЗ с человеком
   - hardhat ↔ область головы
   - vest    ↔ область торса
    │
    ▼
временная логика подтверждения
   - подсчёт streak отсутствия СИЗ
   - длительность > seconds_threshold
   - cooldown между повторами
    │
    ▼
events.csv / events.jsonl (raw)
    │
    ▼
EventConsolidatorV3
   - объединение событий одного эпизода
   - устойчивость к смене track_id
    │
    ▼
consolidated_events.csv  +  processed.mp4
```

Подробное описание модулей: [docs/architecture.md](docs/architecture.md).

## Финальные режимы

### Hardhat

- Детектор: YOLO с тремя классами `person / head / hardhat`.
- Логика: ассоциация каска ↔ область головы, временное подтверждение,
  EventConsolidatorV3.
- Конфиг: `configs/production_hardhat.yaml`.

### Vest

- Детектор: YOLO с прямым классом `NO-Safety Vest`
  (низкий conf-порог + жёсткие требования по треку).
- Логика: ассоциация жилет ↔ область торса, временное подтверждение,
  EventConsolidatorV3.
- Конфиг: `configs/production_vest.yaml`.

## Результаты

| Задача | Конфигурация | TP / FP / FN | Precision | Recall | F1 |
| --- | --- | ---: | ---: | ---: | ---: |
| no_hardhat | proposed pf01 + EventConsolidatorV3 | 13 / 0 / 3 | 1.000 | 0.813 | 0.897 |
| no_vest | old_vest_lowconf_fine + EventConsolidatorV3 | 21 / 6 / 2 | 0.778 | 0.913 | 0.840 |

  (см. [docs/evaluation.md](docs/evaluation.md)).
- Результаты не являются промышленной сертификацией.
- Baseline-сравнения и подробности экспериментов приведены в основной
  части ВКР.

## Сравнение с baseline

В ходе исследования система сравнивалась с несколькими альтернативными
подходами: покадровыми detector-based решениями, SH17, YOLO-World и
CLIP/VLM baseline. Сравнение показало, что событийная логика с
привязкой СИЗ к человеку и консолидацией повторов снижает шум и делает
результат пригодным для анализа на уровне инцидентов. Сами
baseline-скрипты в этот репозиторий не переносились — он содержит только
итоговое решение.

Краткая таблица сравнения приведена в
[docs/experiment_summary.md](docs/experiment_summary.md).

```bash
python -m venv .venv

source .venv/bin/activate
# Windows
.venv\Scripts\activate

pip install -r requirements.txt
```

Файлы весов моделей не входят в репозиторий. Поместите их в каталог

пути в конфиге.

## Веса моделей и демонстрационные видео (Google Drive)

Веса детекторов и демонстрационные видео распространяются отдельно от
репозитория, так как являются крупными бинарными файлами. Архив
размещён на Google Drive:

**[Открыть папку на Google Drive](https://drive.google.com/drive/folders/1YmBQYMUwpaXqmMdpY5acyaaalxJCVmNv)**

В папке находится архив `google_drive_bundle.rar` (около 127 МБ).

### Что внутри архива

Архив содержит файлы, необходимые для запуска системы:

- веса YOLO-детекторов (`.pt`), ожидаемые в каталоге `models/`
  (см. [models/README.md](models/README.md));
- демонстрационные видео для воспроизведения работы pipeline.

Перед использованием рекомендуется сверить состав архива с описанием в
[models/README.md](models/README.md). Если имена файлов в архиве
отличаются от ожидаемых, поправьте поле `model.weights_path` в
соответствующем YAML-конфиге.

### Распаковка

1. Скачайте `google_drive_bundle.rar` по ссылке выше.
2. Распакуйте архив любым подходящим инструментом
   (например, [7-Zip](https://www.7-zip.org/) на Windows
   или `unar` / `unrar` на Linux/macOS).
3. Перенесите распакованные файлы в каталоги проекта:
   - `*.pt` (веса детекторов) → `models/`
     (см. [models/README.md](models/README.md));
   - демонстрационные видео → `input/` (или укажите свой путь
     через параметр `--source` при запуске).
4. Убедитесь, что пути в выбранном YAML-конфиге
   (`configs/production_hardhat.yaml`, `configs/production_vest.yaml`)
   соответствуют именам файлов после распаковки.

После этого можно переходить к разделу «Запуск».

## Запуск

Hardhat-режим:

```bash
python main.py --config configs/production_hardhat.yaml \
    --source input/demo.mp4 --output runs/demo_hardhat
```

Vest-режим:

```bash
python main.py --config configs/production_vest.yaml \
    --source input/demo.mp4 --output runs/demo_vest
```

Пост-обработка готового events.csv консолидатором V3:

```bash
python tools/consolidate_events.py --task hardhat \
    --input runs/demo_hardhat/events.csv \
    --output runs/demo_hardhat/consolidated_events.csv
```

## Выходные файлы

В каталоге, заданном через `--output`, будут созданы:

| Файл | Содержимое |
| --- | --- |
| `processed.mp4` | видео с наложенными bbox, статусами и плашками |
| `events.csv` | сырые события из временной логики |
| `events.jsonl` | то же в JSONL для удобной потоковой обработки |
| `consolidated_events.csv` | итоговые события после `EventConsolidatorV3` |
| `frame_metrics.csv` | покадровая телеметрия (число детекций, треков) |
| `runtime_profile.json` | тайминги стадий pipeline |
