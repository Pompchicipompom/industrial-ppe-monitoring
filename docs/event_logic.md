# Событийная логика

## Что такое «событие нарушения»

Событие нарушения — это запись о факте устойчивого отсутствия требуемого
СИЗ у конкретного человека в видеопотоке. Событие имеет:

- тип (`no_hardhat` или `no_vest`),
- момент фиксации (`frame_idx`, `timestamp_sec`),
- идентификатор трека человека (`person_track_id`),
- bbox человека на момент срабатывания.

Одно физическое нарушение должно соответствовать одному событию.
Кратковременные исчезновения детекции или смена track_id не должны
порождать дубликаты.

## Покадровая логика подтверждения

В `event_logic.py` реализована покадровая логика, использующая две
независимые проверки:

1. **Счётчик подряд идущих кадров без СИЗ** — `consecutive_frames`.
2. **Длительность нарушения в секундах** — `seconds_threshold`.

Событие может быть зафиксировано только если выполнены оба условия
одновременно. Это защищает и от «лишних» событий на короткой паузе
между кадрами с СИЗ, и от ложных событий при очень низкой частоте
инференса.

Дополнительно:

- `cooldown_seconds` — минимальный интервал между двумя событиями
  для одного и того же `track_id`.
- `lock_after_confirm` (только для hardhat) — после подтверждения
  состояния «без каски» оно фиксируется до явного появления каски
  длительностью `hardhat_confirm_frames`.

## TP / FP / FN на event-level

| Тип | Описание |
| --- | --- |
| **TP** | predicted event попал в окно GT-события (`[start_frame, end_frame]`) и не использовался ранее |
| **FP** | predicted event вне всех GT-окон |
| **FN** | GT-событие не было покрыто ни одним predicted event |

FN считается на уровне GT-события, а не на уровне отдельных кадров.
TN на event-level не определяется (см. [evaluation.md](evaluation.md)).

## Cooldown между повторами

После фиксации события для конкретного `track_id` накладывается
cooldown в `cooldown_seconds`. Это нужно, чтобы за одно физическое
нарушение в журнале появлялось ровно одно событие, а не серия повторов
по мере того, как условие подтверждения переоценивается на каждом
последующем кадре.

## Эпизод-консолидация и `EventConsolidatorV3`

Покадровая логика работает в рамках одного `track_id`. На практике
один физический человек может получить несколько `track_id` подряд
(occlusion, выход из кадра и возврат). Чтобы такие случаи не давали
дубликаты:

- `EventConsolidatorV3` собирает все raw-события по видео;
- сортирует их по времени;
- проверяет, попадает ли новое событие в окно времени и пространства
  уже зафиксированного эпизода;
- если попадает — событие сливается с эпизодом и в итоговый журнал
  не добавляется;
- если выпадает — создаётся новый якорь / новое событие.

Совпадение определяется по комбинации:
- временной интервал `same_episode_max_gap_seconds`;
- IoU bbox-якоря и нового события >= `iou_threshold`;
- или центр нового bbox внутри расширенного bbox-якоря
  (`expanded_anchor_bbox_ratio`, `center_distance_ratio`).

История bbox-якорей хранится `history_seconds` секунд, что устойчиво к
небольшим сдвигам тела человека.

## Несколько нарушителей в одном видео

Допускаются:

- один человек = один эпизод нарушения (повторы внутри эпизода
  объединяются);
- несколько разных людей = несколько независимых эпизодов
  (флаг `allow_multiple_tracks_per_video = True` в V3).

Это критично для сцен с несколькими работниками, у части из которых
СИЗ отсутствует.

## Почему повторы одного эпизода объединяются

Серия повторных срабатываний по одному физическому нарушению:

- занижает Precision (каждое повторение засчитается как FP);
- засоряет журнал событий;
- мешает связать событие с конкретным инцидентом.

Эпизод-консолидация даёт ровно одно событие на одно физическое нарушение
и устойчива к «дрейфу» track_id.
